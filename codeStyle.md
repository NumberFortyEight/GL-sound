# Памятка по написанию кода

Короткая, обязательная. Применять к каждому изменению.

## Философия

Меньше кода — меньше багов. Короткий, безопасный, функциональный код с минимальной когнитивной сложностью.
На каждый кусок: «Логика верна? Код короче? Достаточно ли высокоуровнево?»

## Стиль

- Только явные типы — никаких `var`.
- Guard clauses — ранние `return` для невалидных случаев, минимум вложенности. **Действует и внутри лямбд**: `if (!ok) return true;`, а ошибочная ветка идёт плоско следом. Никогда не оборачивай ошибку в `if (bad) { … throw }` — это лишняя вложенность.
- Stream API вместо циклов; method references вместо лямбд-обёрток (`Type::getX`, не `x -> x.getX()`).
- Optional вместо тернарников `?:` и null-проверок — для любых guarded-трансформаций цепочка `Optional`, а не тернарник.
- Вертикальное форматирование цепочек: первый `.` на строке с получателем, каждый следующий `.method()` — на своей строке. Никаких горизонтальных «лапшей» из билдеров/Optional.
- Разворачивание вложенных вызовов `f(g(x))` в последовательность шагов: каждый шаг делает ровно одну операцию, не вкладывай одну в другую. Повторяющуюся трансформацию нормализуй один раз у источника, а не пересчитывай в каждом шаге.
  - Пример (`Optional`): начинай с самого «сырого» источника и вешай каждый вызов отдельным `.map(...)` — `.map(g).map(f)`, а не `Optional.ofNullable(f(g(x)))`.
  - Пример (`Stream`): трансформацию выноси отдельной стадией `.map(...)` (желательно method reference), не прячь её в предикат/терминальную лямбду — `.map(String::trim).allMatch(t -> t.matches(...))`, а не `.allMatch(t -> t.trim().matches(...))`.
- Без комментариев кода. Без лишних пустых строк — один пробел только после внешнего вызова (DB/HTTP).
- Строки — через `"...%s...".formatted(x)`, никогда не конкатенация `"'" + x + "'"`.
- Терминальную операцию приклеивай к закрывающей `}` блочной лямбды: `}).ifPresent(customer::setName)`, а не висящей отдельной строкой `.ifPresent(...)`.
- Без single-use промежуточных переменных — прямой chaining. **Но это правило уступает двум вещам:** (а) переменной, которая *включает* guard (`boolean isDuplicate = …; if (!isDuplicate) return true;` читается лучше, чем вызов внутри `if`); (б) значению, используемому и в логе, и в исключении. В этих случаях — именуй. Иначе оставлять переменную только при 2+ использованиях или для важного доменного концепта.

## Чего НЕ делать

- Без helper-методов на 2-3 строки для предиката/трансформации, используемой 1-2 раза — инлайнить тело лямбды.
- Без dead code «на будущее» — никаких public-методов или Supplier-обёрток, когда реально вызывается одна ветка.
- Без дублирующих тяжёлых fetch'ей в одном методе — взять раз, спроецировать в памяти дважды.

## Архитектура / данные

- Bulk IN-запросы, никаких N+1 — не `ids.forEach(service::deleteById)`; нет bulk-метода — добавить.
- `@Modifying` на каждый кастомный `@Query` DELETE/UPDATE в Spring Data JDBC.
- orgId прокидывать через DTO-носители (`AssemblyData` и т.п.), не перезапрашивать `getWorkInfo`.
- Микросервисные классы НЕ наследуют от KMC — KMC это параллельное зеркало, не родитель. И KMC не трогать без явной просьбы.

## Исключения

- Бросать только доменные исключения — `DomainException` и наследники. Никогда `ResponseStatusException` или HTTP-исключения фреймворка: статус ставит глобальный `@RestControllerAdvice`.
- Сообщение для лога и для исключения — одна строка-переменная (один источник правды), не дублировать текст.

## Логирование

- Короткие `ServiceName methodName key=val` (entry/state/count) через SLF4J-плейсхолдеры `{}`; счётчики — через `list.size()` после forEach.
- Только аннотация `@Slf4j`, никаких других способов. Без `System.out.println()`.
- Когда лог = текст исключения, логируй уже собранную строку (`log.warn(exceptionInfo)`), плейсхолдеры тут не нужны.
